# OpenAI Embeddings Handoff Plan

## Purpose

The live app can call OpenAI directly for query embeddings with `QUERY_EMBEDDING_PROVIDER=openai`, but that only works if the movie vectors stored in Postgres were generated by the same OpenAI embedding model and dimensionality.

The current semantic-search path was originally wired around 384-dimensional movie embeddings generated with `sentence-transformers/all-MiniLM-L6-v2`. OpenAI embeddings and MiniLM embeddings are different vector spaces. Even if both are 384-dimensional, cosine/vector distance between them is not meaningful.

This project needs a fresh set of OpenAI-generated movie embeddings in Postgres before the Vercel app should use OpenAI as its query embedding provider.

## Current App Contract

The app generates a query vector in `src/app/api/lib/queryEmbeddings.js` and compares it against a selected pgvector column in `src/app/api/lib/movieRepository.js`.

Runtime env vars used by the app:

```env
QUERY_EMBEDDING_PROVIDER=openai
OPENAI_API_KEY=<server-side OpenAI API key>
QUERY_EMBEDDING_MODEL=<chosen OpenAI embedding model>
QUERY_EMBEDDING_DIMENSIONS=<must match the Postgres vector column dimension>
SEMANTIC_SEARCH_EMBEDDING_COLUMN=<chosen movie embedding column>
```

The app currently allows these semantic columns:

```text
embedding_semantic
embedding_overall
embedding_thematic
embedding_emotional
embedding_stylistic
embedding_nanogenres
```

If a new column name is preferred, coordinate an app change to add it to the allowlist before launch.

## Recommended Decision

Use `text-embedding-3-small` unless there is a quality reason to pay for larger vectors.

Two viable schema approaches:

1. **Minimal app/schema disruption:** generate OpenAI embeddings with `dimensions=384` and store them in a new `vector(384)` column, for example `embedding_openai_3_small_384`.
2. **Higher-fidelity OpenAI default:** use the model's default dimensionality and add a matching pgvector column. This may require more storage, larger indexes, and an app allowlist update.

The minimal-disruption path is preferred for a quick production rollout because the app already defaults to 384 dimensions and the existing semantic inventory is built around 384-dimensional vectors.

## Required Work

1. Pick the final OpenAI embedding model and dimensions.
   - Recommended: `text-embedding-3-small`
   - Recommended dimensions for low-risk rollout: `384`

2. Add a dedicated Postgres column for the OpenAI vectors.
   - Do not overwrite the current MiniLM vectors until the new search has been evaluated.
   - Suggested column: `embedding_openai_3_small_384 vector(384)`

3. Generate source text for each movie consistently.
   - Use the same source fields for every row.
   - Include enough semantic signal for recommendation/search quality.
   - Suggested input shape:

```text
Title: <title>
Year: <release_year>
Director: <director>
Genres: <genres>
Nanogenres: <nanogenres>
Overview: <overview or review-derived summary>
```

4. Create embeddings with the OpenAI embeddings endpoint.
   - Use the exact same `model` and `dimensions` that the Vercel app will use for query embeddings.
   - Store vectors as pgvector-compatible arrays.
   - Batch and retry safely.
   - Track failures and rerun only missing rows.

5. Add an HNSW cosine index on the new column after backfill.

Example:

```sql
create index concurrently if not exists movies_embedding_openai_3_small_384_hnsw_idx
on public.movies
using hnsw (embedding_openai_3_small_384 vector_cosine_ops)
where embedding_openai_3_small_384 is not null;
```

6. Report coverage and readiness.

```sql
select
  count(*) as total_movies,
  count(*) filter (where embedding_openai_3_small_384 is not null) as embedded_movies,
  round(
    100.0 * count(*) filter (where embedding_openai_3_small_384 is not null) / nullif(count(*), 0),
    2
  ) as coverage_pct
from public.movies;
```

7. Hand the final column/model/dimension values back to the app project.

Expected Vercel env vars for the recommended path:

```env
QUERY_EMBEDDING_PROVIDER=openai
OPENAI_API_KEY=<server-side OpenAI API key>
QUERY_EMBEDDING_MODEL=text-embedding-3-small
QUERY_EMBEDDING_DIMENSIONS=384
SEMANTIC_SEARCH_EMBEDDING_COLUMN=embedding_openai_3_small_384
```

The app must be updated to allow `embedding_openai_3_small_384` before using that column.

## Acceptance Tests

Before the app switches to OpenAI provider in production, verify:

1. The new vector column exists and has the expected dimension.
2. At least 95% of movies have non-null OpenAI embeddings, preferably 100%.
3. The selected column has an HNSW cosine index.
4. Query embeddings generated with the same OpenAI model/dimensions can be cast to pgvector and compared with `<=>`.
5. Sample semantic searches return plausible movies:
   - `slow meditative science fiction`
   - `surreal hollywood mystery`
   - `warm animated coming of age`
   - `tense paranoid political thriller`
   - `melancholy romantic drama`
6. The app team confirms `/api/movies?searchMode=semantic&title=<query>&limit=8` works in Preview with the OpenAI env vars.

## Cost Note

Embedding query cost with OpenAI should be very low for user searches. The one-time backfill for a few thousand movies should also be small. The main requirement is not cost; it is vector-space consistency between stored movie embeddings and live query embeddings.
